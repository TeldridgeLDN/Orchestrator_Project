**Version:** 1.0
**Date:** 2025-11-10
**Status:** Draft for Implementation

---

## Executive Summary

This PRD defines a **Partnership-Enabled Scenario System** that transforms the orchestrator from a simple project manager into an intelligent AI workflow development platform. The system introduces "scenarios" as first-class user-facing abstractions that compile into orchestrator primitives (skills, workflows, commands, hooks, MCPs).

**Core Innovation:** The AI acts as a **consultant partner**, not an order taker. It researches alternatives, challenges assumptions, validates feasibility, and continuously optimizes deployed workflows.

---

## Problem Statement

### Current Limitations

**From Orchestrator PRD:**
- Users must manually decompose business processes into Skills, Workflows, Agents, MCPs, Commands, and Hooks
- No guidance on which primitive to use for a given scenario
- No alternative exploration ("is there a better way?")
- No feasibility validation before implementation
- No optimization suggestions for deployed workflows

**User Pain Points:**
1. "I don't know if I should create a Skill, Agent, or MCP for my use case"
2. "I described my workflow, but there might be simpler approaches I don't know about"
3. "I want to capture client requirementsâ€”should I use Google Forms or something else?"
4. "My workflow has been running for a month. Can it be improved?"

---

## Solution: Three-Layer Architecture

### Layer 1: Scenario Layer (User Mental Model)
**Location:** `~/.claude/scenarios/`
**Purpose:** User-facing abstraction for business processes
**Format:** Declarative YAML with steps, dependencies, triggers

**Example:**
```yaml
scenario:
  name: client_intake
  description: "Capture client requirements before building landing site"
  trigger:
    type: hybrid
    command: "/client-intake"
    keywords: ["new client", "requirements"]
  steps:
    - send_form
    - wait_completion
    - analyze_requirements
    - create_project
```

### Layer 2: Global Layer (Implementation)
**Location:** `~/.claude/`
**Purpose:** Generated implementation components
**Components:** Skills, Commands, Hooks, MCP configs

**Generated from scenarios**

### Layer 3: Project Layer (Codebases)
**Location:** `<project>/.claude/`
**Purpose:** Project-specific behavior (unchanged from PRD)

### 2. Scenario Definition Format

**File:** `~/.claude/scenarios/<name>.yaml`

**Schema:**
```yaml

  name: string
  version: string
  description: string
  category: string
  
  trigger:

    command: string (slash command)
    keywords: [strings] (auto-activation)
    schedule: cron (if scheduled)
  
  steps:
    - id: string
      type: external_api|monitor|notification|ai_analysis|orchestrator_call
      action: string
      mcp_required: string
      inputs: [strings]
      outputs: [strings]
      conditions: [conditions]
      timeout: duration
  
  dependencies:
    mcp_servers: [strings]
    skills: [strings]
    agents: [strings]
  
  generates:
    - global_skill: string
    - slash_command: string
    - hook_pattern: string
  
  design_decisions:
    - decision: string
      date: ISO8601
      rationale: string
      alternatives_considered:
        - option: string

          rejected_because: string
      trade_offs:
        pros: [strings]
        cons: [strings]
  
  potential_improvements:
    - suggestion: string
      impact: low|medium|high
      complexity: low|medium|high
      estimated_effort: string
      status: pending|approved|rejected|implemented
```

### 3. Partnership Levels

**Setting:** `~/.claude/config.json` â†’ `partnership_level`

**Options:**
1. **order_taker** - No alternatives, just build what user says
2. **clarifier** - Ask questions, don't challenge
3. **advisor** - Research alternatives, recommend best (DEFAULT)
4. **partner** - Deep exploration, proactive optimization

**User Control:**
```bash
# Global setting
claude config set partnership_level advisor

# Per-scenario override
claude scenario create --partnership-level partner
```

### 4. Required MCP Servers

scenario_builder needs these MCPs to be a partner:

**1. Research MCP** (e.g., Perplexity)
- Query: "Best practices for client requirements capture 2025"
- Purpose: Stay current, research alternatives

**2. MCP Registry MCP** (custom)
- Query: "What MCPs exist for Airtable?"
- Purpose: Know what's buildable

**3. API Documentation MCP**
- Query: "Does Airtable API support webhooks?"
- Purpose: Validate feasibility

**4. Cost Calculator MCP** (custom)
- Query: "Monthly cost: Typeform Pro + Airtable Plus"
- Purpose: Inform decisions with real costs

**5. Community Scenarios MCP** (future)
- Query: "Show similar client intake scenarios"
- Purpose: Learn from proven patterns

### 5. Feasibility Checker Sub-Agent

**Location:** `~/.claude/agents/feasibility_checker/`

**Purpose:** Parallel validation of proposed integration approaches

**System Prompt:**
```markdown
You validate technical feasibility of proposed integrations.

Check:
1. MCP availability
2. API capabilities (webhooks, rate limits)
3. Authentication methods
4. Known blockers
5. Cost requirements

**Output Format:**
[FEASIBILITY_SCORE: X/10] Brief analysis. Blockers: [list]
```

**Usage:**
- scenario_builder spawns 3 instances in parallel
- Each validates one alternative approach
- Results inform recommendations in ~10 seconds

## CLI Commands

### Scenario Management

```bash
# Create scenario interactively (partner-guided)
claude scenario create

# Create from template
claude scenario create --template client-intake

# Deploy scenario (scaffold all components)
claude scenario deploy <name>

# List all scenarios
claude scenario list

# Show scenario details
claude scenario show <name>

# Edit and redeploy
claude scenario edit <name>

# Validate scenario definition
claude scenario validate <name>

# Remove scenario
claude scenario remove <name>

# Optimize deployed scenario
claude scenario optimize <name>

# Research alternatives for existing scenario
claude scenario explore <name>
```

### Partnership Configuration

```bash
# Set partnership level
claude config set partnership_level advisor

# Get current partnership level
claude config get partnership_level

# Per-scenario override
claude scenario create --partnership-level partner
```

## Key Workflows

### explore_alternatives.md

**Purpose:** Research and propose 3+ implementation approaches

**Process:**
1. Extract core need (not proposed solution)
2. Research using MCP: "best practices for X 2025"
3. Generate 3 options: Simplest, User's idea, Advanced
4. Validate feasibility (spawn 3 sub-agents in parallel)
5. Analyze trade-offs (pros/cons, complexity, cost)
6. Recommend best option with reasoning
7. Present structured comparison table

### test_feasibility.md

**Purpose:** Validate technical possibility before recommending

**Checks:**
1. MCP availability
2. API capabilities (webhooks, rate limits)
3. Authentication methods
4. Known blockers (deprecated APIs)
5. Cost requirements (plan tiers)

**Output:** Feasibility score (0-10) + blockers list

### optimize_scenario.md

**Purpose:** Suggest improvements to deployed scenarios

**Process:**
1. Read scenario YAML + usage logs
2. Analyze performance metrics
3. Research current best practices
4. Identify optimization opportunities
5. Rank by impact/complexity/risk
6. Present top 3 suggestions
7. Offer to implement highest ROI

### compare_options.md

**Purpose:** Deep dive trade-off analysis

**When:** User requests "tell me more about Option A vs B"

**Output:**
- Detailed feature comparison
- Sample workflows for each
- Monthly cost breakdown
- Implementation effort estimates
- Maintenance burden analysis
- Risk assessment

### proof_of_concept.md

**Purpose:** Test approach before full deployment

**Process:**
1. Create minimal test scenario
2. Validate API connections
3. Test data flow
4. Identify blockers early
5. Show demo to user
6. Confirm approach or pivot
